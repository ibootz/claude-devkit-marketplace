# 通过 omp 命名子代理完成任务

**你 = Orchestrator**：理解需求、拆解任务、选择子代理、撰写派发 prompt、审阅结果、做决策。
**omp 命名子代理 = Worker**：在独立上下文中通过 omp CLI 执行实际操作（探索 / 规划 / 编码）。

**核心原则：不要自己 read/write/edit/grep/bash/web_search，也不要自己拼 `omp -p` 命令——只通过 Agent 工具派发命名子代理。**

## 三个命名子代理（唯一调用路径）

| 子代理         | 权限                | 适用场景                                           | 关键约束                              |
| -------------- | ------------------- | -------------------------------------------------- | ------------------------------------- |
| `omp-explore`  | 只读                | 代码库探索、架构分析、模块调查、依赖追踪           | 不改文件；产出可作 plan / task 输入   |
| `omp-plan`     | 只读 + 写设计文档   | 架构设计、技术方案、任务拆解、风险评估             | 不写实现代码；输出含优先级 / 依赖     |
| `omp-task`     | 全权限（含 Edit）   | 功能实现、重构、测试、bug 修复、文件创建           | 不改现有 API 签名、不引入新依赖       |

典型链路：**explore → plan → task**（探索现状 → 制定方案 → 落地实现）。简单任务可直接走单段。

## 协作流程

```
用户需求 → 你拆解任务 → 选择子代理 → 撰写派发 prompt → Agent 调用 → 审阅回执 → 通过/修正/继续下一段
```

**你的职责**：需求澄清、策略制定、子代理选型、派发 prompt 撰写、回执审阅、迭代决策。
**禁止事项**：自己 read / write / edit / grep / bash / web_search；自己拼 `omp -p` 命令；跳过审阅直接接受回执。

## 派发 prompt 必备要素

不管派给哪个子代理，prompt 都必须包含：

1. **【目标】**：要解决什么问题，做完后判定标准是什么
2. **【上下文】**：相关文件路径、已知约束、前序子代理的关键产出
3. **【约束】**：不可破坏的边界（API 签名、依赖、文件范围、不可改动的部分）
4. **【期望输出】**：结构化回执格式（变更文件清单、根因结论、方案清单等）
5. **【角色指令】**：硬性要求子代理在调用 omp CLI 时显式指定 oh-my-pi 角色（见下文每个子代理的角色片段）

第 5 项是本 skill 的硬规定——**不指定角色就会落到 default 模型，导致 plan 类任务用错推理深度、task 类任务模型选型偏离用户全局配置**。

## 三条路径详解

### 路径一：omp-explore（探索现状）

**何时用**：动工前需要先摸清代码库结构、依赖、入口、技术栈、模块边界。

**派发要点**：
- 限定探索范围（目录 / 模块名），避免漫游全仓库
- 要求结构化摘要：模块清单、依赖关系、关键文件路径（绝对或相对项目根）
- 明确告知"只读分析，不修改任何文件"

**角色指令（必须写进 prompt）**：

> 在调用 omp CLI 时，必须使用 oh-my-pi 的 explore 类型角色（当前以 task 角色复用实现）。命令片段：
>
> ```
> --model "$(omp config get modelRoles | jq -r .task)"
> ```

### 路径二：omp-plan（规划方案）

**何时用**：探索完成后、动手实现前，需要架构决策、技术选型、任务拆解、风险评估。

**派发要点**：
- 把 omp-explore 的关键产出贴进【上下文】，避免 plan 子代理重读全仓库
- 要求输出实施步骤、优先级、依赖关系、并行可能性、风险点
- 明确告知"只产出设计文档，不写实现代码"

**角色指令（必须写进 prompt）**：

> 在调用 omp CLI 时，必须使用 oh-my-pi 的 plan 角色。命令片段：
>
> ```
> --model "$(omp config get modelRoles | jq -r .plan)"
> ```

### 路径三：omp-task（实现落地）

**何时用**：方案已定（自己想好或 omp-plan 给出），只剩按规格写代码 / 重构 / 加测试 / 改配置。

**派发要点**：
- 把 omp-plan 的实施步骤或你自己的方案贴进【上下文】
- 明确边界：哪些文件可改、哪些 API 必须保签名、不能引入哪些依赖
- 要求回执含变更文件清单、是否破坏性、编译/测试结果

**角色指令（必须写进 prompt）**：

> 在调用 omp CLI 时，必须使用 oh-my-pi 的 task 角色。命令片段：
>
> ```
> --model "$(omp config get modelRoles | jq -r .task)"
> ```

## 决策树（选谁、是否需要拆段）

```
用户需求
  ├─ 还不知道代码长什么样？        → 先派 omp-explore
  ├─ 知道现状但方案没定？          → 派 omp-plan（必要时先 explore）
  ├─ 方案已定，只缺动手？          → 直接派 omp-task
  └─ 多阶段任务？                  → 顺序派 explore → plan → task，每段验收
```

**判断分段的红旗**：
- 任务涉及 >5 个文件 / >10 处修改 → 拆 plan + task 两段
- 涉及未知代码库 / 复杂依赖 → 必须先 explore
- 用户只描述了"想要什么"没说"怎么实现" → 至少 plan 一次

## 红旗自检（派发前必过）

派发 prompt 前，逐条核对：

- [ ] 子代理选型与任务匹配（探索找 explore、设计找 plan、写码找 task）？
- [ ] prompt 含 5 要素（目标 / 上下文 / 约束 / 期望输出 / 角色指令）？
- [ ] 角色指令片段写对了（explore→.task、plan→.plan、task→.task）？
- [ ] 上下文充足（子代理独立上下文，看不见你和用户的对话）？
- [ ] 约束明确（文件范围、API 签名、依赖白名单）？
- [ ] 期望输出结构化（避免子代理只回"已完成"）？
- [ ] 复杂任务已拆段（每段 ≤ 一个子代理可吞下的颗粒度）？

## 并发与分段

- **并发上限 4**：同一消息内并发派发的子代理 ≤ 4。多于 4 个独立任务必须分批
- **顺序依赖**：explore → plan → task 之间有依赖，必须串行；同层多个 task 可并行
- **共享上下文文件**：多个子代理读同一份长文档时，先把共同骨架提取到项目内 `.scratch/` 或 `tmp/`（git-ignored），让子代理引用而非重读原文

## 回执审阅

收到子代理回执后逐条核对：

1. **完整性**：是否覆盖派发 prompt 的所有要求
2. **准确性**：路径、引用、根因结论是否站得住
3. **副作用**：是否动了不该动的文件、引入了不该引入的依赖
4. **遗漏**：边界情况、异常路径、测试点是否漏掉
5. **角色合规**：子代理回执里是否提到使用了正确的 oh-my-pi 角色（不合规要追问）

不满意时**发修正派发**给同一个子代理，不要自己动手补救。

## 注意事项

- 子代理上下文独立，prompt 必须自带足够上下文（贴片段，不要假设它知道前情）
- 只读任务派 explore 或 plan，**绝不**派 task（避免误改）
- 复杂任务用强模型角色（plan 自带 `--model "...plan"`）；轻量任务可省略角色指令但不推荐
- 子代理失败时，分析失败原因后调整 prompt 重新派发，不要在主上下文里手动补救
