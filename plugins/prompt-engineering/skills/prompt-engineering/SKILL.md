# 提示词工程 - Prompt Engineering

为 Agent 编写命令、钩子、技能，或为子代理和其他 LLM 交互优化提示词，包括改进 LLM 输出和设计生产级提示模板。

## 核心能力

### 1. Few-Shot Learning（少样本学习）

通过示例而非规则教学模型。包含 2-5 个输入-输出对展示期望行为。用于一致格式化、特定推理模式或边界情况处理。

**示例：**

```markdown
从支持工单中提取关键信息：

输入："我的登录不工作，一直收到 403 错误"
输出：{"issue": "authentication", "error_code": "403", "priority": "high"}

输入："功能请求：在设置中添加暗黑模式"
输出：{"issue": "feature_request", "error_code": null, "priority": "low"}

现在处理："无法上传大于 10MB 的文件，超时"
```

### 2. Chain-of-Thought（思维链）

在最终答案前请求逐步推理。添加"让我们一步步思考"（零样本）或包含示例推理轨迹（少样本）。

**示例：**

```markdown
分析此错误报告并确定根本原因。

逐步思考：

1. 预期行为是什么？
2. 实际行为是什么？
3. 最近有什么变化可能导致这个问题？
4. 涉及哪些组件？
5. 最可能的根本原因是什么？

错误："昨天部署缓存更新后，用户无法保存草稿"
```

### 3. 提示词优化

通过测试和改进系统性提升提示词。从简单开始，测量性能（准确性、一致性、token 使用），然后迭代。

**示例：**

```markdown
版本 1（简单）："总结这篇文章"
→ 结果：长度不一致，遗漏要点

版本 2（添加约束）："用 3 个要点总结"
→ 结果：结构更好，但仍缺少细节

版本 3（添加推理）："识别 3 个主要发现，然后总结每个"
→ 结果：一致、准确、捕获关键信息
```

### 4. 模板系统

构建可重用的提示结构，包含变量、条件部分和模块化组件。

**示例：**

```python
# 可重用代码审查模板
template = """
审查这段 {language} 代码的 {focus_area}。

代码：
{code_block}

提供以下方面的反馈：
{checklist}
"""

# 使用
prompt = template.format(
    language="Python",
    focus_area="安全漏洞",
    code_block=user_code,
    checklist="1. SQL 注入\n2. XSS 风险\n3. 身份验证"
)
```

### 5. 系统提示设计

设置在对话中持续存在的全局行为和约束。定义模型的角色、专业水平、输出格式和安全指南。

**示例：**

```markdown
系统：你是一位专注于 API 设计的高级后端工程师。

规则：

- 始终考虑可扩展性和性能
- 默认建议 RESTful 模式
- 立即标记安全问题
- 提供 Python 代码示例
- 使用早返回模式

响应格式：

1. 分析
2. 建议
3. 代码示例
4. 权衡
```

## 关键模式

### 渐进式披露

从简单提示开始，仅在需要时添加复杂性：

1. **Level 1**: 直接指令 - "总结这篇文章"
2. **Level 2**: 添加约束 - "用 3 个要点总结，关注关键发现"
3. **Level 3**: 添加推理 - "阅读文章，识别主要发现，然后用 3 个要点总结"
4. **Level 4**: 添加示例 - 包含 2-3 个输入-输出对示例

### 指令层次

```
[系统上下文] → [任务指令] → [示例] → [输入数据] → [输出格式]
```

## 最佳实践

1. **具体明确**：模糊提示产生不一致结果
2. **展示而非告知**：示例比描述更有效
3. **广泛测试**：在多样化、代表性输入上评估
4. **快速迭代**：小改变可能产生大影响
5. **监控性能**：在生产中跟踪指标
6. **版本控制**：像对待代码一样对待提示词
7. **记录意图**：解释为什么这样构建提示词

## Agent 提示最佳实践

基于 Anthropic 官方最佳实践。

### 上下文窗口

上下文窗口是语言模型在生成新文本时可以回顾和引用的全部文本量加上它生成的新文本。这是模型的"工作记忆"。

- 渐进式 token 累积：随着对话推进，每条用户消息和助手响应都累积在上下文窗口中
- 线性增长模式：上下文使用随每轮线性增长，之前的轮次完全保留
- 200K token 容量：总可用上下文窗口代表存储对话历史和生成新输出的最大容量

### 简洁是关键

上下文窗口是公共资源。你的提示、命令、技能与 Claude 需要知道的其他所有内容共享上下文窗口。

**默认假设**：Claude 已经非常聪明

只添加 Claude 还没有的上下文。挑战每条信息：

- "Claude 真的需要这个解释吗？"
- "我可以假设 Claude 知道这个吗？"
- "这段话值得它的 token 成本吗？"

**好例子：简洁**（约 50 tokens）：

````markdown
## 提取 PDF 文本

使用 pdfplumber 提取文本：

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

**坏例子：过于冗长**（约 150 tokens）：

```markdown
## 提取 PDF 文本

PDF（便携式文档格式）文件是包含文本、图像和其他内容的常见文件格式。
要从 PDF 中提取文本，你需要使用库。有许多可用于 PDF 处理的库，
但我们推荐 pdfplumber，因为它易于使用并能处理大多数情况...
```

### 设置适当的自由度

根据任务的脆弱性和可变性匹配具体程度。

**高自由度**（基于文本的指令）：

- 多种方法都有效
- 决策取决于上下文
- 启发式指导方法

**中等自由度**（带参数的伪代码或脚本）：

- 存在首选模式
- 可接受一些变化
- 配置影响行为

**低自由度**（特定脚本，很少或没有参数）：

- 操作脆弱且容易出错
- 一致性至关重要
- 必须遵循特定顺序

## 说服原则

LLM 对与人类相同的说服原则有响应。研究表明说服技术使合规率翻倍（33% → 72%）。

### 七大原则

1. **权威**：专业、凭证、官方来源
   - 使用命令式语言："你必须"、"绝不"、"始终"
   - 用于纪律执行技能、安全关键实践

2. **承诺**：与先前行动、声明一致
   - 要求公告："宣布技能使用"
   - 强制明确选择："选择 A、B 或 C"

3. **稀缺性**：时间限制或有限可用性的紧迫性
   - 时间限制要求："在继续之前"
   - 顺序依赖："在 X 之后立即"

4. **社会证明**：符合他人所做或被认为正常的事
   - 通用模式："每次"、"始终"
   - 失败模式："没有 Y 的 X = 失败"

5. **统一**：共享身份、"我们感"
   - 协作语言："我们的代码库"、"我们是同事"
   - 用于协作工作流、建立团队文化

6. **互惠**：回报收到的好处的义务
   - 谨慎使用 - 可能感觉操纵性

7. **喜欢**：与我们喜欢的人合作的偏好
   - **不要用于合规** - 与诚实反馈文化冲突

## 原则组合

| 提示类型  | 使用                   | 避免       |
| --------- | ---------------------- | ---------- |
| 纪律执行  | 权威 + 承诺 + 社会证明 | 喜欢、互惠 |
| 指导/技术 | 适度权威 + 统一        | 重度权威   |
| 协作      | 统一 + 承诺            | 权威、喜欢 |
| 参考      | 仅清晰                 | 所有说服   |

## 伦理使用

**合法**：

- 确保遵循关键实践
- 创建有效文档
- 防止可预测的失败

**非法**：

- 为个人利益操纵
- 制造虚假紧迫性
- 基于内疚的合规

**测试**：如果用户完全理解，这种技术是否符合用户的真正利益？
