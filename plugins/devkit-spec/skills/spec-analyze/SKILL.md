## argument-hint
<需求描述> 例如：实现用户认证模块，支持邮箱密码登录和 JWT token

## 流程

### 阶段 1：提问式需求引导

1. 根据用户输入的需求，识别关键不确定点
2. 使用 `ask_user_question` 工具进行结构化提问
3. 问题类型：
   - **边界**：功能范围、包含/不包含什么
   - **约束**：技术栈限制、性能要求、安全要求
   - **优先级**：MVP vs 完整功能
   - **非功能需求**：性能、可用性、可维护性

### 阶段 2：代码调研分析（Subagent 并行）

启动 2-3 个 subagent 并行探索不同维度，节省主进程 token：

**Subagent 1: 空间侦察（当前代码结构）**
- 使用 `ace-tool` (Auggie Context MCP) 的 `search_context` 搜索相关代码
- 扫描现有 API、DB Schema、核心类
- 识别相关模块和文件

**Subagent 2: 时间侦察（历史演进）**
- 使用 `history-detective` skill 调查相关文件的历史变更
- 识别曾经被回滚的方案或已知的坑

**Subagent 3: 依赖分析（影响面评估）**
- 分析依赖关系图
- 评估变更影响范围

### 阶段 3：多 LLM 并行分析汇总

将调研结果分发给多个分析维度：

1. **技术可行性分析**：评估实现难度和技术选型
2. **架构影响分析**：评估对现有架构的影响
3. **风险评估**：识别潜在风险和缓解方案

### 阶段 4：澄清现状

基于代码调研和多 LLM 分析结果：
1. 向用户澄清现状发现
2. 识别硬性约束、历史教训、红线
3. 如发现新的不确定点，返回阶段 1 继续引导

### 阶段 5：生成 Spec 文档

当需求边界清晰、约束明确时，生成 `./docs/specs/<slug>-<timestamp>/spec.md`

**spec.md 必须包含**：
- **范围**：功能边界、包含/不包含
- **现状**：相关代码现状、约束
- **方案**：推荐实现方案
- **验收标准**：可验证的验收条件

## 错误处理

- 若用户输入过于模糊，最多引导 3 轮，仍不清晰则建议用户先明确需求
- 若代码调研失败（如目录不存在），明确告知用户并建议检查项目结构
- 若历史调研发现冲突方案，必须暂停并与用户确认

## 核心原则

1. **不做假设**：遇到不确定点必须提问
2. **事实优先**：接口/字段定义以代码现状为准
3. **历史否决**：若某方案曾被回滚，必须规避或提供解决手段
4. **节省 token**：使用 subagent 并行调研，避免主进程阻塞

## 输出路径

`./docs/specs/<slug>-<timestamp>/spec.md`

- `<slug>`：需求简短标识（如 `auth`、`dark-mode`）
- `<timestamp>`：`YYYYMMDD-HHMMSS` 格式

## 使用示例

```
/spec-analyze 实现用户认证模块，支持邮箱密码登录和 JWT token
```
