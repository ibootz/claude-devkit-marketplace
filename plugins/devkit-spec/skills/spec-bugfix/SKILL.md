## argument-hint

<问题描述或 issue 编号> 例如：/spec-bugfix #123 登录时提示 "Invalid credentials"

## 适用场景

- 用户已明确反馈某个缺陷，需要定位并修复
- 原因未知，需要先复现和诊断
- 原因已知，但希望用统一流程完成隔离修复与验证
- 当前分支正在进行其他开发，希望避免互相干扰
- 已有 Spec 文档，需要基于 Spec 进行缺陷修复

## 模式判断

- **诊断优先**：症状明确、根因不明，先复现和定位
- **快速修复**：根因和改动范围已知，直接实施最小修复

## 流程

### 阶段 1：问题收敛

1. 提取 issue 编号、报错信息、影响范围和复现线索
2. 判断属于前端、后端、数据、配置还是环境问题
3. 如果信息不足，先列出最少量的澄清问题
4. 如有相关 Spec 文档，对照 Spec 确认预期行为与实际偏差

### 阶段 2：隔离策略

1. 判断是否需要 `git worktree`
2. 满足以下任一条件时，优先创建隔离 worktree：
   - 当前工作区已有未完成改动
   - 需要多人并行处理不同问题
   - 修复风险较高，可能需要反复试验
3. 若无需隔离，可直接在当前工作区处理，但要明确风险

### 阶段 3：复现与诊断

1. 先寻找最小复现路径，记录触发条件
2. 优先查看报错堆栈、日志、最近相关改动和配置差异
3. 并行检查高概率方向：
   - 代码路径与数据流
   - 依赖或环境差异
   - 相关测试、断言与边界情况
4. 汇总证据，确认根因是否能解释全部症状

### 阶段 4：实施修复

1. 采用最小改动原则修复根因
2. 若修复比预期复杂，暂停并重新评估影响面
3. 同步补齐必要测试，避免同类问题回归

### 阶段 5：验证与收尾

1. 运行最相关的测试或验证脚本
2. 手动复测关键场景和边界情况
3. 记录修复结论：
   - 根因
   - 修改点
   - 验证结果
   - 仍存在的限制或后续建议
4. 如果使用了 worktree，修复完成后清理隔离目录和分支

## 核心原则

1. 先复现，再修复；无法复现时不要草率改代码
2. 根因证据要能闭环解释症状
3. 修复优先最小化，不顺手重构无关代码
4. 变更后必须验证，至少覆盖主路径和一条边界路径

## 错误处理

- 若无法复现，要求补充日志、输入样例、环境信息或时间窗口
- 若问题依赖外部系统，先隔离本地可验证部分，再标记外部阻塞
- 若需要大范围重构才能根治，明确区分"止血修复"和"彻底治理"

## 使用示例

```
# 原因未明，先诊断再修复
/spec-bugfix #123 用户反馈登录时提示 "Invalid credentials"

# 原因已知，直接快速修复
/spec-bugfix 修复 Safari 下登录按钮点击无响应的问题
```
