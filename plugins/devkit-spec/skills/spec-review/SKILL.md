## argument-hint
[spec目录路径] 例如：docs/specs/auth-20260129-143000/（默认审查最近编辑的spec）

## 模式

- **无参数**：审查最近编辑的 spec 目录
- **指定路径**：审查特定规范目录

## 流程

### 阶段 1：收集审查材料

1. 读取 spec.md（规范文档）
2. 读取 tasks.md（任务清单，如有）
3. 读取 impl.md（实现日志，如有）
4. 收集实现代码（根据 tasks.md 和 impl.md 中的文件列表）

### 阶段 2：多 LLM 并行审查

分发给多个分析维度并行审查：

**LLM 1: 规范完整性审查**
- spec.md 结构是否完整
- 逻辑是否自洽
- 验收标准是否可验证

**LLM 2: 实现一致性审查**
- 代码实现是否符合 spec
- 文件变更是否与 tasks.md 对应
- 是否有遗漏的任务

**LLM 3: 质量与风险审查**
- 代码质量评估
- 潜在问题和风险
- 性能和安全考量

### 阶段 3：汇总分析结果

整合各 LLM 的审查意见：
1. 去重和分类问题
2. 按严重程度排序
3. 识别冲突意见并协调

### 阶段 4：生成综合报告

输出格式：

```markdown
# Spec 审查报告

## 概览
- **审查对象**: docs/specs/auth-20260129-143000/
- **审查时间**: 2026-01-29 16:00
- **审查范围**: spec.md + tasks.md + impl.md + 实现代码

## 规范完整性
- [x] spec.md 包含所有必需章节
- [x] 逻辑自洽，无矛盾
- [ ] 验收标准部分不够明确（详见问题列表）

## 实现一致性
- [x] 代码实现符合 spec
- [x] 文件变更与 tasks.md 对应
- [ ] T3 的实现缺少错误处理（详见问题列表）

## 代码质量评估
- **整体评分**: 8/10
- **优点**: 结构清晰，命名规范
- **问题**: 部分函数过长，建议拆分

## 问题列表

### 高优先级
1. **T3 Repository 缺少错误处理**
   - 位置: `src/repository/user.ts:45`
   - 问题: 数据库查询失败未处理
   - 建议: 添加 try-catch 和错误日志

### 中优先级
2. **验收标准不够具体**
   - 位置: spec.md §4.1
   - 问题: "性能良好"无法量化验证
   - 建议: 明确响应时间指标，如 "< 200ms"

### 低优先级
3. **代码注释不足**
   - 建议: 为核心函数添加 JSDoc 注释

## 遗留问题
- 无

## 总体结论
**建议通过**，但需修复高优先级问题。
```

## 审查清单

- [ ] spec.md 是否包含所有必需章节
- [ ] tasks.md 的 DAG 是否无循环依赖
- [ ] impl-log.md 中的文件变更是否与 tasks.md 对应
- [ ] 实现代码是否符合 spec 中的验收标准
- [ ] 是否有遗留问题未解决
- [ ] 代码质量、性能、安全性评估

## 错误处理

- 若 spec 目录不存在，提示用户检查路径
- 若缺少关键文件（spec.md），提示补充
- 若审查过程中发现严重问题，暂停并通知用户

## 输出

综合审查报告（Markdown 格式，输出到聊天）

## 使用示例

```
# 审查最近的 spec
/spec-review

# 审查指定目录
/spec-review docs/specs/auth-20260129-143000/
```
