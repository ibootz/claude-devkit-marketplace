# 发现你的未知（Discover Your Unknowns）

一套与 Claude 协作的**任务生命周期方法论**。源自 Anthropic 官方博客 *A field guide to Claude Fable 5: Finding your unknowns*（Thariq Shihipar）。

## 核心心智模型

- **地图 vs 疆域**：地图 = 你给 Claude 的提示词、技能、上下文；疆域 = 工作真正发生的地方（代码库、现实约束）。两者的差距就是**未知（unknowns）**。
- 工作量越大，Claude 遇到的未知越多；每个未知处，Claude 都只能基于对你意图的**最佳猜测**做决定。
- **减少并为未知做规划，是"智能体编程"的核心技能**——且这是一项可以通过与 Claude 协作而持续提升的技能。
- 仅靠"提前规划"不够：未知可能潜伏在实现深处，甚至会指向"应该换一种完全不同的解法"。因此这是一个贯穿**实现前 / 实现中 / 实现后**的迭代过程。

## 四类未知（拆解任务时先过一遍）

| 类型 | 含义 | 对应动作 |
|------|------|----------|
| 已知的已知 Known Knowns | 我清楚要什么，能写进提示词 | 直接写清楚 |
| 已知的未知 Known Unknowns | 我知道自己还没想清楚的点 | 访谈 / 头脑风暴 |
| 未知的已知 Unknown Knowns | 显而易见到不会写下、但看到就认得（尤指"看到才知道好坏"的标准） | 原型 / 参考 |
| 未知的未知 Unknown Unknowns | 完全没考虑过、根本没意识到的知识 | 盲点扫描 |

> 顶尖的智能体编程者未知较少（与代码库和模型行为深度同步），但**他们仍会主动假定存在未知并为之预留空间**。

## 指令的平衡（关键原则）

- **太具体** → Claude 会死板执行你的指令，即便转向（pivot）更合适也不转。
- **太含糊** → Claude 会套用不一定契合你任务的"行业最佳实践"。
- 破解之道：**不要只下命令，要交代背景**——告诉 Claude 你的起点、你在这个问题/代码库上的经验、你的思路进展，把它当**思考伙伴（thought partner）**，借它"极快检索代码库与互联网 + 从失败中快速迭代"的能力，帮你更快发现未知。

## 何时启用本方法论（编排逻辑）

作为 Claude，在以下信号出现时**主动**提议或直接进入对应模式，而不是抓起第一条相关命令就干：

- 任务在**陌生领域** / 代码库的新区域，或用户自陈"不熟悉这块" → 先 **盲点扫描**
- 目标是**看到才知道好坏**的东西（UI/设计/文案/数据可视化） → 先 **头脑风暴与原型**
- 需求存在**模糊点、且答案会影响架构** → 先 **访谈**
- 用户"说不清但有样例/现成实现" → 索要 **参考**（源代码优先）
- 准备动手、且涉及数据模型/类型接口/UX 流程等**易变决策** → 先出 **实现计划**
- 进入实际编码、可能撞上边界情况 → 全程维护 **实现笔记**
- 交付需要他人**理解与批准** → 产出 **提案/讲解**
- 长会话后用户需**确认变更内容** → 做 **测验**

多个信号同时命中时，按"实现前 → 中 → 后"的时序推进；每一步的产出（规格、原型、计划、笔记）作为下一步的输入传入。

## 八个实践模式

> 每个模式给出「何时用 / 怎么做 / 示例 prompt」。示例 prompt 直接取自原文，可按当前任务改写占位内容。每个模式都有**同名独立 skill**（标在标题后），可通过 `/<skill 名>` 直接调用，也可由本 skill 按编排逻辑路由过去；独立 skill 里有更完整的执行细则，深入执行某模式时优先调用对应 skill。
>
> 其中 **头脑风暴** 与 **访谈** 两个模式：若当前可用技能列表中存在 `superpowers:brainstorming`（用户已安装 superpowers 插件），优先委派该 skill 执行（访谈场景叠加"优先问会改架构的问题"约束）；不存在时才按本插件自定义指令执行。

### 实现前

#### 1. 盲点扫描 Blind Spot Pass — `/blind-spot-pass`
- **何时用**：陌生领域，大量"未知的未知"；不知道该问什么、好的标准长什么样、历史做过什么、有哪些坑。
- **怎么做**：明确用"盲点扫描""未知的未知"这样的字眼；先给 Claude 关于"你是谁、你已知什么"的背景，让它据此决定如何与你开始协作，并把挖到的未知**讲解**给你。
- **示例 prompt**：
  - "我正在为代码库添加一个新的认证提供方，但我对这个代码库里的认证模块一无所知。你能做一次盲点扫描，帮我理清相关的未知的未知，并帮助我更好地向你提问吗？"
  - "我不知道什么是调色（color grading），但我需要给这段视频调色。你能教我理解自己关于调色的未知的未知，好让我能更好地向你提问吗？"

#### 2. 头脑风暴与原型 Brainstorms & Prototypes — `/brainstorm`
- **何时用**：领域里有大量"未知的已知"——那些你只有看到才知道如何定义的标准（视觉设计、布局、方向选择）。在原型阶段说出它们**远比在实现阶段发现更便宜**（功能/规格的小改动会造成代码实现的巨大差异，且让智能体回退改动更难）。
- **怎么做**：让 Claude 用**假数据 + 单个 HTML 文件**快速做可反应的原型（不接后端、不维护多余状态）；或为同一产物给出**多个风格迥异的方向**供你挑；几乎每个编码会话都以探索/头脑风暴开场，防止范围设得太窄或太宽。
- **示例 prompt**：
  - "我想为这些数据做一个仪表盘，但我没有视觉审美，也不知道有哪些可能性。给我做一个 HTML 页面，包含 4 个风格迥异的设计方向，好让我做出反应。"
  - "在接通任何东西之前，用假数据做一个单独的 HTML 文件，模拟新的编辑器工具栏。我想在你动真正的应用之前先对布局做出反应。"
  - "这是我大致的问题：用户在引导之后流失。搜索代码库，头脑风暴出 10 个我们可以介入的地方，从最省成本到最有雄心排列。我会告诉你哪些让我有共鸣。"

#### 3. 访谈 Interviews — `/interview`
- **何时用**：头脑风暴后仍有未知/含糊，需要 Claude 反过来问你。
- **怎么做**：给 Claude 关于问题的背景以引导提问；要求**一次一个问题**，并**优先问那些"你的回答会改变架构"的问题**。
- **示例 prompt**：
  - "就任何含糊之处一次一个问题地访谈我，优先提出那些我的回答会改变架构的问题。"

#### 4. 参考 References — `/reference`
- **何时用**：你无法详细描述想要什么（缺乏对应语言，或复杂到说清要很久）。
- **怎么做**：给一个**参考**。图表/文档/图片都行，但**绝对最好的参考是源代码**——把 Claude 指向那个文件夹并告诉它要找什么，即使是另一种语言写的；相比截图，源码能提供关于标记与结构远为丰富的细节。
- **示例 prompt**：
  - "vendor/rate-limiter 里的这个 Rust crate 实现了我想要的退避（backoff）行为。读一读它，并在我们的 TypeScript API 客户端里重新实现相同的语义。"

#### 5. 实现计划 Implementation Plan — `/impl-plan`
- **何时用**：你觉得准备好实现了，动手前先审阅计划。
- **怎么做**：让 Claude 出一份计划，**把你最可能改动的决策放在最前面**（数据模型改动、新的类型接口、任何面向用户的部分），**把机械性重构沉到最底部**（那部分信任 Claude）。这样计划能主动**浮现你真正需要改的东西**。建议用 HTML 输出便于阅读。
- **示例 prompt**：
  - "用 HTML 写一份实现计划，但要以我最可能调整的决策开头：数据模型的改动、新的类型接口，以及任何面向用户的部分。把机械性的重构放在最底部，那部分我信任你。"

### 实现中

#### 6. 实现笔记 Implementation Notes — `/impl-notes`
- **何时用**：计划敲定、开始实际编码后。无论规划多充分，总有"未知的未知"潜伏，智能体可能在代码里撞上边界情况需要改换思路。
- **怎么做**：满意计划后**新开一个会话**，把规划阶段的产物（规格、原型）作为提示词传入（获得全新上下文窗口 + 全部已编译信息）；让 Claude Code 维护一个临时的 `implementation-notes.md`（或 `.html`）记录所做决策，遇到被迫偏离计划的边界情况时**选保守方案、记在 "Deviations" 下、继续推进**，供下一次尝试汲取经验。
- **示例 prompt**：
  - "维护一个 implementation-notes.md 文件。如果你遇到某个边界情况，迫使你偏离计划，就选择保守的方案，把它记录在 'Deviations' 下，然后继续推进。"

### 实现后

#### 7. 提案与讲解 Pitches & Explainers — `/pitch`
- **何时用**：交付需要拿到支持与批准。
- **怎么做**：在最终文档里构建提案（pitch）与说明（explainer）材料——当评审者带着和你当初相同的未知开始时**加速理解**；当专家想看到你已考虑了他们会预料的未知与常见故障点时**加速批准**。可打包成一份能直接丢进 Slack 的文档，开头放演示 GIF。
- **示例 prompt**：
  - "把原型、规格说明和实现笔记打包成一份文档，让我能直接丢到 Slack 上争取支持。开头先放演示 GIF。"

#### 8. 测验 Quizzes — `/quiz`
- **何时用**：长会话后，Claude 可能做了比你意识到的多得多的改动；只读 diff 只能得到粗浅理解（很多行为依赖已有代码路径）。
- **怎么做**：给足上下文后，让 Claude 就本次变更**对你进行测验**；只有**完美通过测验后才合并**。
- **示例 prompt**：
  - "我想确保自己理解这次变更中发生的一切。给我一份关于这些变更的 HTML 报告，让我可以带着上下文、直觉去阅读理解，包括做了什么等等，并在末尾附上一个我必须通过的、关于这些变更的测验。"

## 收束原则

- 模型越强，用对方法能取得的成果越多。**长周期任务返回了错误结果**，往往意味着你需要花更多时间界定未知，或做一份"能让你和 Claude 边推进边调整"的实现计划。
- 每一次讲解、头脑风暴、访谈、原型和参考，都是**在代价高昂、难以修复之前，低成本地发现你此前不知道的东西**。
- 一句话行动指南：**开始下一个项目时，先请 Claude 帮你找出你的未知。**

## 反模式（避免）

- 抓起第一条看起来相关的命令就执行，跳过澄清未知。
- 指令一步到位地过度具体，堵死更优转向。
- 在实现深处才第一次暴露"未知的已知"，造成昂贵返工。
- 长会话结束只读 diff 就 merge，未做测验/讲解确认。
