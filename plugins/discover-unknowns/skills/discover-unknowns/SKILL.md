# 发现你的未知（Discover Your Unknowns）

一套与 Claude 协作的**发现未知**方法论。源自 Anthropic 官方博客 *A field guide to Claude Fable 5: Finding your unknowns*（Thariq Shihipar）。

## 核心心智模型

- **地图 vs 疆域**：地图 = 你给 Claude 的提示词、技能、上下文；疆域 = 工作真正发生的地方（代码库、现实约束）。两者的差距就是**未知（unknowns）**。
- 工作量越大，Claude 遇到的未知越多；每个未知处，Claude 都只能基于对你意图的**最佳猜测**做决定。
- **减少并为未知做规划，是"智能体编程"的核心技能**——且这是一项可以通过与 Claude 协作而持续提升的技能。
- 仅靠"提前规划"不够：未知可能潜伏在实现深处，甚至会指向"应该换一种完全不同的解法"。因此发现未知贯穿任务始终，不是一次性动作。

## 四类未知（拆解任务时先过一遍）

| 类型 | 含义 | 对应手法 |
|------|------|----------|
| 已知的已知 Known Knowns | 我清楚要什么，能写进提示词 | 直接写清楚 |
| 已知的未知 Known Unknowns | 我知道自己还没想清楚的点 | 访谈 / 头脑风暴（`/brainstorm`） |
| 未知的已知 Unknown Knowns | 显而易见到不会写下、但看到就认得（尤指"看到才知道好坏"的标准） | 原型（`/brainstorm`）/ 参考（下文） |
| 未知的未知 Unknown Unknowns | 完全没考虑过、根本没意识到的知识 | 盲点扫描（下文） |

> 顶尖的智能体编程者未知较少（与代码库和模型行为深度同步），但**他们仍会主动假定存在未知并为之预留空间**。

## 指令的平衡（关键原则）

- **太具体** → Claude 会死板执行你的指令，即便转向（pivot）更合适也不转。
- **太含糊** → Claude 会套用不一定契合你任务的"行业最佳实践"。
- 破解之道：**不要只下命令，要交代背景**——告诉 Claude 你的起点、你在这个问题/代码库上的经验、你的思路进展，把它当**思考伙伴（thought partner）**，借它"极快检索代码库与互联网 + 从失败中快速迭代"的能力，帮你更快发现未知。

## 路由（信号 → 手法）

作为 Claude，在以下信号出现时**主动**提议或直接进入对应手法，而不是抓起第一条相关命令就干：

| 信号 | 手法 | 入口 |
|------|------|------|
| 陌生领域 / 代码库新区域 / 用户自陈"不熟悉这块" | 盲点扫描 | 本 skill 内嵌（见下） |
| 目标"看到才知道好坏"（UI/设计/文案/可视化），或需求含糊需收敛 | 头脑风暴（发散）/ 访谈（收敛） | `/brainstorm` |
| 用户"说不清但有样例/现成实现" | 参考 | 本 skill 内嵌（见下） |
| 长会话大量变更后要确认理解再合并 | 测验 | `/quiz` |

多个信号同时命中时按顺序推进：盲点扫描 → 头脑风暴/访谈 → 参考，每一步的产出作为下一步的输入。

> **边界**：未知收敛后的**实施规划与执行**（实现计划、任务拆解、按计划编码）不属于本方法论，交给规范驱动类工作流（如本市场的 devkit-spec：`/spec-analyze` → `/spec-tasks` → `/spec-impl`）。唯一延伸到实现中的原则：**撞上迫使偏离既定方案的边界情况 = 新暴露的未知**——选保守方案、显式记录偏离并告知用户，不要静默变通。

## 内嵌手法一：盲点扫描（Blind Spot Pass）

**何时用**：用户处在陌生领域，存在大量"未知的未知"——不知道该问什么、好的标准长什么样、历史做过什么、有哪些坑。

**怎么做**：

1. 先向用户确认背景：你是谁、已知什么、目标是什么（若上下文已有则不重复问）。
2. 扫描相关代码库/领域知识，列出用户**很可能没意识到**的关键未知：隐含约束、历史决策、常见坑、好坏标准、可选方案空间。
3. 逐条**讲解**——不只列名词，要说明"为什么这对你的任务重要、忽略它会怎样"。
4. 结尾给出"你现在可以问出的更好的问题"清单，帮用户学会更好地提问。
5. **这一步只暴露和讲解未知，不做实现。**

**示例 prompt**（原文）：

- "我正在为代码库添加一个新的认证提供方，但我对这个代码库里的认证模块一无所知。你能做一次盲点扫描，帮我理清相关的未知的未知，并帮助我更好地向你提问吗？"
- "我不知道什么是调色（color grading），但我需要给这段视频调色。你能教我理解自己关于调色的未知的未知，好让我能更好地向你提问吗？"

## 内嵌手法二：参考（References）

**何时用**：用户无法详细描述想要什么（缺乏对应语言，或复杂到说清要很久），但有现成样例——参考实现、喜欢的组件、另一语言的库。

**怎么做**：

1. 图表/文档/图片都可作参考，但**绝对最好的参考是源代码**——相比截图，源码能提供关于标记与结构远为丰富的细节。主动向用户索要源码级参考。
2. 读懂参考后，**先复述你理解到的语义**（行为、边界、取舍），与用户对齐。
3. 在当前项目**复刻语义而非细节**：即使参考是另一种语言/框架写的，重新实现的是相同的行为语义，不是逐行翻译。

**示例 prompt**（原文）：

- "vendor/rate-limiter 里的这个 Rust crate 实现了我想要的退避（backoff）行为。读一读它，并在我们的 TypeScript API 客户端里重新实现相同的语义。"

## 收束原则

- 模型越强，用对方法能取得的成果越多。**长周期任务返回了错误结果**，往往意味着你需要花更多时间界定未知。
- 每一次盲点扫描、头脑风暴、访谈、原型和参考，都是**在代价高昂、难以修复之前，低成本地发现你此前不知道的东西**。
- 一句话行动指南：**开始下一个项目时，先请 Claude 帮你找出你的未知。**

## 反模式（避免）

- 抓起第一条看起来相关的命令就执行，跳过澄清未知。
- 指令一步到位地过度具体，堵死更优转向。
- 在实现深处才第一次暴露"未知的已知"，造成昂贵返工。
- 长会话结束只读 diff 就 merge，未做测验确认（该做 `/quiz`）。
