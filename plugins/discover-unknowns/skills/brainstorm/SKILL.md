# 头脑风暴与访谈（Brainstorm & Interview）

围绕用户给出的目标（随调用传入的参数或当前对话上下文），在需求模糊阶段交替使用两种手法暴露未知：

- **发散**：Claude 产出多方向原型/方案 → 换用户的反应（暴露"未知的已知"——看到才知道好坏的标准）。
- **收敛**：Claude 一次一个问题地访谈用户 → 换用户的答案（澄清"已知的未知"——影响架构的模糊点）。

两者不是二选一：典型流程是先发散给用户看方向，再就用户反应中的模糊点收敛提问，如此往复直到关键未知收敛。

## 优先委派 superpowers（若已安装）

先检查当前可用技能列表中是否存在 `superpowers:brainstorming`（用户已安装 [superpowers](https://github.com/obra/superpowers) 插件；其 brainstorming 本身就融合了方案探索与一次一问的对话流程）：

- **存在** → 通过 Skill 工具调用 `superpowers:brainstorming`，把用户的目标原样传入，按其流程执行。同时叠加本方法论的两条差异化约束：可视化产物用**假数据单 HTML 原型、多方向**供反应；提问时**优先问"回答会改变架构"的问题**。
- **不存在** → 按下方自定义指令执行。

## 自定义指令（fallback）

### 发散：头脑风暴与原型

- **多方向探索**：为同一产物给出多个**风格迥异**的方向（默认 3-4 个）供用户反应。若是可视化产物（UI/仪表盘/布局），产出**单独的 HTML 文件、用假数据**呈现——不接后端、不维护多余状态。
- **范围头脑风暴**：搜索相关代码库/上下文，头脑风暴多个可介入的方案，**从最省成本到最有雄心排列**，由用户挑选有共鸣的。
- 主动指出你发现的高价值但用户可能错过的方向，并提醒范围是否太窄或太宽。

### 收敛：访谈

1. **一次只问一个问题**，等用户回答后再问下一个；严禁一次抛出问题清单。
2. **优先提出"用户的回答会改变架构/技术选型/数据模型"的问题**；纯偏好、可后置的细节往后放或跳过。
3. 每个问题给出你倾向的默认答案与理由，让用户可以只回"就按你说的"或纠正你。
4. 不要问从已有背景里能推断的东西。
5. 关键未知基本收敛时**主动停止**，输出**结论摘要**：已澄清的决策 + 仍待定的点。

### 共同边界

- 价值在于**尽早、廉价地暴露未知**，避免拖到实现阶段才浮现——那时功能/规格的小改动会造成代码实现的巨大差异，且回退更难。
- 不接通真实系统，不做超出"可反应原型"所需的工作。
- 产出后**等用户反应**，未知收敛前不进入实现。

## 原文示例 prompt（供理解意图）

- "我想为这些数据做一个仪表盘，但我没有视觉审美，也不知道有哪些可能性。给我做一个 HTML 页面，包含 4 个风格迥异的设计方向，好让我做出反应。"
- "在接通任何东西之前，用假数据做一个单独的 HTML 文件，模拟新的编辑器工具栏。我想在你动真正的应用之前先对布局做出反应。"
- "这是我大致的问题：用户在引导之后流失。搜索代码库，头脑风暴出 10 个我们可以介入的地方，从最省成本到最有雄心排列。我会告诉你哪些让我有共鸣。"
- "就任何含糊之处一次一个问题地访谈我，优先提出那些我的回答会改变架构的问题。"
