# 实现计划（Implementation Plan）

为用户给出的工作（随调用传入的参数或当前对话上下文）写一份实现计划。

## 结构要求（顺序很重要）

1. **开头放用户最可能调整的决策**：数据模型的改动、新的类型/接口定义、任何面向用户（UX/API/文案）的部分。每条给出你的选择、理由，以及"若改这里会连带影响什么"。
2. **中间**放核心实现步骤与依赖顺序。
3. **最底部**放机械性重构/样板改动——这部分用户信任你，不必展开细节。
4. 显式列出计划里仍存在的**未知/假设**：标明哪些需要用户拍板、哪些你会在实现时按保守方案处理。

## 输出与边界

- 用 **HTML 文件**输出便于阅读审阅（用户另有偏好时从其偏好）。
- 计划的目的是**主动浮现用户真正需要改的东西**，不是流水账。
- 先不要开始写实现代码，等用户审完计划。

## 原文示例 prompt（供理解意图）

- "用 HTML 写一份实现计划，但要以我最可能调整的决策开头：数据模型的改动、新的类型接口，以及任何面向用户的部分。把机械性的重构放在最底部，那部分我信任你。"
