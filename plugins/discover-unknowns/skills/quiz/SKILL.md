# 测验（Quiz）

用户想确保自己**完全理解本次变更中发生的一切**（随调用传入的参数指定范围，或基于当前会话/diff），再决定是否合并。

## 产出：HTML 报告 + 测验

1. **报告部分**：带上下文和直觉地讲清楚——做了什么改动、为什么这么做、它如何依赖/影响已有的代码路径、有哪些非显而易见的行为。（只读 diff 无法体现这些，很多行为取决于已有代码路径，所以必须讲清依赖关系。）
2. **测验部分**（放在报告末尾）：设计一组用户**必须答对**的题目，覆盖关键决策、行为变化、潜在故障点。题目应检验真实理解，而非背诵。
3. 先出报告和题目，等用户作答；答完后判分并指出理解偏差之处。

## 硬约束

- **只有用户完美通过测验，才进行合并。** 在通过前不要执行 merge/push 等操作。

## 原文示例 prompt（供理解意图）

- "我想确保自己理解这次变更中发生的一切。给我一份关于这些变更的 HTML 报告，让我可以带着上下文、直觉去阅读理解，包括做了什么等等，并在末尾附上一个我必须通过的、关于这些变更的测验。"
